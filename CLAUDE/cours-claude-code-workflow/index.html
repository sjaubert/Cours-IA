Premiers Pas

Installer et lancer Claude Code

Qu'est-ce que Claude Code ?

Claude Code est un assistant IA de developpement en ligne de commande (terminal). Contrairement aux chatbots classiques utilises dans un navigateur, Claude Code s'integre directement dans votre environnement de travail : il peut lire vos fichiers, proposer des modifications, executer des commandes et comprendre la structure complete de votre projet.

Imaginez un expert consultant qui s'installe a votre bureau, ouvre votre dossier de travail, lit tous vos documents et vous aide en connaissant deja tout le contexte — plutot que de tout lui reexpliquer a chaque rencontre.

Installation et demarrage

Prerequis : Node.js version 18 ou superieure installee sur votre machine.

# Etape 1 : Installer Claude Code globalement
curl -fsSL https://claude.ai/install.sh | bash

# Etape 2 : Se placer dans votre dossier projet
cd votre-projet

# Etape 3 : Lancer Claude Code
claude

# Etape 4 : Initialiser la memoire du projet
/init

Que fait la commande /init ?

La commande /init est une etape cruciale. Elle analyse l'integralite de votre projet (fichiers, structure, technologies) et cree automatiquement un fichier CLAUDE.md qui sera le point de depart de la memoire de Claude pour toutes les sessions futures.

Toujours executer /init en premier dans un nouveau projet. Claude analysera votre code, vos dependances, et generera un fichier memoire pertinent que vous pourrez affiner.

Comprendre CLAUDE.md

La memoire persistante de votre projet

CLAUDE.md est le fichier de memoire persistante de Claude pour votre projet. Il est charge automatiquement au debut de chaque session — vous n'avez pas a tout reexpliquer a chaque fois que vous recommencez a travailler.

C'est comme le cahier de bord d'un technicien de maintenance. Chaque fois qu'il prend son poste, il lit le cahier pour connaitre l'etat de la machine, les problemes recents, et les procedures a suivre. CLAUDE.md joue ce role pour l'IA.

Structure recommandee

QUOIPOURQUOICOMMENT

Arborescence du projetDecisions de designWorkflows
Architecture generaleContraintes metierPoints d'attention (Gotchas)

Exemple concret : Projet de gestion de production

# Projet : MonApp
# API REST FastAPI + SPA React + PostgreSQL

## Objectif
Application de suivi de production pour l'atelier d'usinage.
Permet aux operateurs de saisir les donnees de controle qualite.

## Stack technique
- Backend : Python FastAPI (API REST)
- Frontend : React (interface utilisateur)
- Base de donnees : PostgreSQL

## Commandes essentielles
npm run dev        # Serveur de developpement
npm run test       # Tests automatiques
npm run lint       # Qualite du code

## Architecture
/app     → pages Next.js (pages de l'interface)
/lib     → utilitaires partages (fonctions communes)
/prisma  → Schema DB et migrations

## Regles de travail
- Toujours ecrire les commentaires en français
- Valider les donnees cote serveur avant insertion
- Ne jamais stocker de mots de passe en clair

## Points d'attention
- La connexion PostgreSQL necessite le VPN interne
- Les tests peuvent prendre 2-3 minutes

- Formations → inscription → validation budget → convocation

- Conserver un historique des modifications

Hierarchie Memoire

Les 4 niveaux de contexte

Claude peut avoir plusieurs fichiers CLAUDE.md a differents niveaux. Ils sont tous charges simultanement, du plus global au plus specifique. Le fichier le plus local affine le contexte sans ecraser le contexte global.

~/.claude/CLAUDE.md ← Niveau 1 : Global (tous vos projets) ↓ /monorepo/CLAUDE.md ← Niveau 2 : Racine du depot ↓ /monorepo/.claude/CLAUDE.md ← Niveau 3 : Config locale (ignoree par Git) ↓ /monorepo/frontend/CLAUDE.md ← Niveau 4 : Sous-dossier (contexte limite)

Regles d'or

  • Gardez chaque fichier sous 200 lignes : Au-dela, la memoire se charge trop lentement et le contexte se dilue.
  • Les fichiers de sous-dossier ajoutent du contexte : Ils ne remplacent pas, ils completent.
  • N'ecrasez jamais le contexte parent : Coherence entre fichiers globaux et locaux.

Exemple pratique : 3 niveaux de CLAUDE.md

# ~/.claude/CLAUDE.md (votre profil global)
- Langue : toujours repondre en français
- Style : commentaires en français, variables en anglais
- Jamais de donnees sensibles dans les exemples
# /projet-usine/CLAUDE.md (specifique au projet)
Ce projet gere les donnees de controle qualite de la ligne 3.

Technologies : Python + SQLite + Pandas

Regles metier :
- Les valeurs hors tolerance doivent declencher une alerte
- Conserver 3 ans d'historique (obligation reglementaire)
# /projet-usine/rapports/CLAUDE.md (sous-dossier rapports)
Ce sous-dossier genere les rapports PDF mensuels.
Utiliser ReportLab. Format : A4 portrait, logo en en-tete.

Bonnes Pratiques

Travailler efficacement avec Claude

1

Executez /init d'abord

Initialisez avant de commencer. Laissez Claude analyser le projet, puis affinez ce qu'il a genere.

2

Soyez precis

"Ameliore les performances de calculer_moyenne() avec NumPy" plutot que "Ameliore ce code".

3

Documentez les pieges

Les contraintes non-evidentes : "le serveur de test est lent le mardi", "ne pas utiliser la lib X".

4

Referencez avec @

Utilisez @CLAUDE.md pour mentionner des fichiers specifiques dans vos demandes.

5

Regles de workflow

Documentez votre facon de travailler : branches Git, validations, tests avant deploiement.

6

Memoire concise

50 lignes pertinentes valent mieux que 500 lignes fouillis. Revoyez et epurez regulierement.

7

Commitez sur Git

Incluez CLAUDE.md dans Git pour que toute l'equipe beneficie du meme contexte partage.

Exemple : fichier commands/deploy.md

# Commande : Deployer en production

## Description
Lance le processus de deploiement complet vers le serveur de production.

## Etapes
1. Verifier que tous les tests passent (npm run test)
2. Construire l'application (npm run build)
3. Verifier les variables d'environnement (NODE_ENV=production)
4. Envoyer sur le serveur (rsync -avz dist/ serveur:/var/www/app)
5. Redemarrer le service (sudo systemctl restart monapp)
6. Verifier que l'application repond (http://monserveur/health)

## En cas d'erreur
Si l'etape 4 echoue : verifier la connexion VPN au reseau interne.
Contacter l'administrateur systeme : poste 2847.

Les Skills

Le superpouvoir de Claude

Un Skill (competence) est un guide Markdown que Claude auto-invoque via le langage naturel. Quand vous demandez quelque chose, Claude cherche automatiquement s'il existe un Skill correspondant et l'utilise comme guide d'expert.

Les Skills sont comme des fiches de procedures dans un classeur d'atelier. Quand un operateur doit effectuer une tache specifique (changer un outil, calibrer une machine), il consulte la fiche correspondante. Claude fait de meme — automatiquement.

Deux types de Skills

P

Skill de projet

Partage avec toute l'equipe via Git.

.claude/skills/<nom>/SKILL.md

U

Skill personnel

Pour vous seul, sur tous vos projets.

~/.claude/skills/<nom>/SKILL.md

Structure d'un SKILL.md

---
name: nom-du-skill
description: Description PRECISE de ce que fait ce skill
              ← ESSENTIEL pour l'auto-activation !
allowed-tools: Read, Grep, Bash, Write
---

# Titre du Skill

## Quand utiliser ce skill
[Conditions d'activation]

## Procedure
[Etapes detaillees]

## Exemples
[Exemples concrets]

Le champ description est essentiel pour l'auto-activation. Claude lit cette description pour decider si le skill correspond a votre demande. Une description vague signifie un skill jamais utilise.

Exemple 1 : Skill de revue de code

---
name: code-review
description: Effectue une revue de code professionnelle en verifiant
              la lisibilite, la securite, les performances et
              le respect des bonnes pratiques
allowed-tools: Read, Grep
---

# Revue de Code

## Checklist de verification
- Les noms de variables sont explicites
- Les fonctions font une seule chose
- Les cas d'erreur sont geres
- Pas de donnees sensibles en dur (mots de passe, cles API)
- Les commentaires expliquent le "pourquoi", pas le "comment"

## Format du rapport
Pour chaque probleme trouve :
- Fichier et ligne : localisation precise
- Severite : Critique / Important / Mineur
- Description : ce qui pose probleme
- Suggestion : comment corriger

Exemple 2 : Skill de messages de commit Git

---
name: commit-messages
description: Formule des messages de commit Git clairs, structures et
              en français selon la convention Conventional Commits
allowed-tools: Bash
---

# Messages de Commit

## Convention
[TYPE]: Resume court en français (max 72 caracteres)

## Types disponibles
- feat: nouvelle fonctionnalite
- fix: correction de bug
- docs: documentation uniquement
- refactor: amelioration sans changement de comportement

## Exemples
feat: ajouter la validation des donnees du formulaire conges
fix: corriger le calcul des jours ouvres lors des ponts

Exemple 3 : Testing Patterns (Modeles de test)

---
name: testing-patterns
description: Jest testing patterns. Utilise describe + it + modele AAA.
              Utilise des mocks d'usine (factory mocks)
allowed-tools: Read, Grep, Glob
---

# Modele de Test AAA
# Arrange / Act / Assert

## Structure
describe('NomDeLaFonction', () => {
  it('doit retourner la bonne valeur', () => {
    // Arrange : preparer les donnees
    const donnees = [10, 20, 30];

    // Act : executer
    const resultat = calculerMoyenne(donnees);

    // Assert : verifier
    expect(resultat).toBe(20);
  });
});

Configuration des Hooks

Automatiser la securite et le controle

Un Hook est un callback deterministe : un script qui s'execute automatiquement a des moments precis de l'interaction avec Claude — independamment de ce que vous demandez.

Les Hooks sont comme les procedures de securite dans une usine : des controles qui s'executent automatiquement a des etapes definies, independamment de l'operateur. Avant de demarrer la machine (PreToolUse), apres son arret (PostToolUse), ou pour une notification.

Les 3 types de Hooks

TypeQuand ?Utilisation typique
PreToolUse Avant qu'un outil soit utilise Verifications de securite
PostToolUse Apres qu'un outil a ete utilise Nettoyage, journalisation
Notification Pour envoyer des alertes Notifier sur Teams / Slack

Configuration dans .claude/settings.json

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",  ← s'active avant TOUTE commande Bash
        "hooks": [
          {
            "type": "command",
            "command": "scripts/securite.sh",
            "timeout": 5  ← timeout 5 secondes
          }
        ]
      }
    ]
  }
}

Codes de sortie (Exit codes)

CodeSignificationExemple
0Autoriser l'actionTout est OK, continuer
2Bloquer l'actionOperation non autorisee

Exemple : Script de securite

#!/bin/bash
# scripts/securite.sh
# Vérifie que Claude ne modifie pas des fichiers critiques

FICHIER_CIBLE="$1"
FICHIERS_PROTEGES=("config/production.env" "secrets/" "backup/")

for protected in "${FICHIERS_PROTEGES[@]}"; do
    if [[ "$FICHIER_CIBLE" == *"$protected"* ]]; then
        echo "ERREUR : Modification interdite sur $protected"
        exit 2  ← Bloquer l'action
    fi
done

exit 0  ← Autoriser l'action

Permissions & Securite

Controler ce que Claude peut faire

Claude ne peut acceder qu'aux ressources que vous lui autorisez explicitement. La configuration des permissions se fait dans .claude/settings.json.

C'est comme les droits d'acces par badge dans un immeuble de bureaux. Une carte peut ouvrir certaines portes (autorisees) mais pas d'autres (refusees). On ne donne pas les cles de toute l'usine a tout le monde.

Structure des permissions

{
  "permissions": {
    "allow": [
      "Read:*",            ← Lire tous les fichiers
      "Bash:git:*",        ← Executer toutes les commandes git
      "Write:*:*:*.md"    ← Ecrire uniquement les .md
    ],
    "deny": [
      "Read:env:*",        ← INTERDIT : variables d'environnement
      "Bash:sudo:*"        ← INTERDIT : commandes administrateur
    ]
  }
}

Exemple : Politique de securite pour une PME

{
  "permissions": {
    "allow": [
      "Read:*",
      "Bash:npm:*",
      "Bash:git:*",
      "Write:src:*",
      "Write:docs:*"
    ],
    "deny": [
      "Read:*.env",
      "Read:secrets:*",
      "Bash:sudo:*",
      "Bash:rm:-rf:*",
      "Write:config/production:*"
    ]
  }
}
AutoriseInterdit
Lire tous les fichiers du projetLire les fichiers .env (mots de passe)
Lancer npm (scripts, packages)Acceder au dossier secrets
Utiliser git (commit, push, pull)Utiliser sudo (droits admin)
Ecrire dans src/ et docs/Supprimer des dossiers entiers

Architecture en 4 Couches

Comment Claude fonctionne en profondeur

Claude Code fonctionne selon une architecture en 4 couches superposees, chacune avec un role precis et complementaire :

L1

CLAUDE.md — Contexte et regles persistants

Charge a chaque session. Ne change pas entre les conversations.

Cahier de bord de l'equipe
L2

Skills — Packs de connaissances auto-invoques

Active uniquement quand pertinent. Expertise specialisee a la demande.

Fiches de procedures metier
L3

Hooks — Passerelles de securite et automatisation

S'executent a chaque evenement defini. Independants de Claude.

Capteurs de securite
L4

Agents — Sous-agents avec leur propre contexte

Taches complexes deleguees. Contexte propre et independant.

Sous-traitants specialises

Synergie des couches — Exemple concret

Voici comment les 4 couches collaborent lors d'une simple demande :

Vous demandez : "Deploie la nouvelle version en production"

→ L1 (CLAUDE.md) :
  Charge le contexte : stack technique, serveur de prod, credentials
  
→ L2 (Skill docker-deploy) :
  Auto-invoque le skill de deploiement Docker
  Suit la procedure definie pas a pas
  
→ L3 (Hook PreToolUse) :
  Verifie que les tests passent avant tout deploiement
  Bloque si un test echoue (exit code 2)
  
→ L4 (Agent security-reviewer) :
  Analyse le code a deployer pour detecter des vulnerabilites
  Rapport genere en parallele

Workflow Quotidien

La routine optimale avec Claude Code

1

Ouvrir le projet

cd mon-projet && claude — Claude charge automatiquement CLAUDE.md et retrouve le contexte.

2

Activer le Mode Plan — Shift + Tab + Tab

Claude presente un plan d'action avant d'executer. Il reflechit avant d'agir. Vous validez le plan, puis il travaille.

3

Decrire votre intention

Etre precis et concret. Exemple : "Ajoute un formulaire de validation des donnees de controle qualite. La valeur doit etre dans la plage [0.95, 1.05] et afficher un message d'erreur rouge si ce n'est pas le cas."

4

Acceptation automatique — Shift + Tab

Active l'acceptation automatique : Claude peut executer les operations approuvees sans vous demander a chaque fois.

5

Compresser le contexte — /compact

Apres une longue session, /compact resument les echanges pour liberer de l'espace memoire. Comme une synthese de reunion en cours de seance.

6

Rewind — Esc Esc

Si Claude s'ecarte de ce que vous vouliez, double-Esc revient en arriere pour reformuler. L'equivalent d'un Ctrl+Z intelligent.

7

Committer frequemment

git commit -m "feat: ajouter la validation du formulaire QC" — Chaque fonctionnalite terminee = un commit. Sauvegarde et possibilite d'annuler.

8

Une session par fonctionnalite

Pour garder la memoire de contexte propre, commencez une nouvelle session Claude pour chaque nouvelle fonctionnalite.

Explication de la Pensee Etendue

La pensee etendue (touche Tab) permet a Claude de raisonner plus longuement et en profondeur avant de repondre. C'est la difference entre demander la reponse immediate a un expert, ou lui donner 15 minutes pour reflechir au tableau blanc. La deuxieme option produit une reponse plus reflechie — mais prend plus de temps.
