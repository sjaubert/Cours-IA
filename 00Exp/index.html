Mise au point critique avant de commencer

Trois avis tranchés, dont deux corrigent des documents qui circulent.

Le guide « Gemma 4 » est en grande partie inventé

Gemma 4 n'existe pas en juin 2026. La dernière famille est Gemma 3 (mars 2025) : tailles 270M, 1B, 4B, 12B, 27B. Sont fabriqués : le « 26B-A4B MoE », le « 31B dense », l'alerte « CUDA 13.2 corruption ». Aucune source.

À garder comme cas d'étude pour ton activité « Illusion de Vérité » : un document peut avoir l'air d'un guide technique sérieux et halluciner ses chiffres. Réflexe à enseigner : vérifier le nom d'un modèle sur la source primaire ollama.com/library.

« Ollama est-il indispensable, y a-t-il plus léger ? »

Fausse prémisse. Ollama, LM Studio, Jan et llama.cpp tournent sur le même moteur (llama.cpp), à 5 % de perf près. Le runner ne consomme pas tes ressources, le modèle chargé en mémoire les consomme.

Tu as déjà Ollama : garde-le. Ta vraie variable d'ajustement, c'est la taille et la quantification du modèle.

AlternativeQuand la préférer
LM StudioInterface graphique pour débutants, démo grand public sans terminal.
JanConfidentialité maximale, open source MIT, zéro télémétrie. Argument souveraineté.
llama.cpp brutMontrer la mécanique bas niveau. Pédagogique mais aride.

La tension à assumer : hétérogène + agent

Sur des portables sans GPU dédié, un agent autonome a besoin d'un modèle qui appelle des outils en JSON valide. Les petits modèles sur CPU produisent le JSON malformé décrit comme un « bug » dans les guides. Ce n'est pas un réglage, c'est une limite structurelle.

Conséquence : paliers 1 et 2 pour tous. Palier 3 sur ta machine, en démonstration.

Le matériel : ce qui compte vraiment

Trois chiffres, par ordre d'importance : 1. VRAM (mémoire GPU), 2. RAM système, 3. CPU. Si le modèle tient en VRAM, c'est rapide. Sinon il déborde en RAM et s'effondre.

Règle 4-bit (la norme) : environ 0,7 Go par milliard de paramètres, plus une marge de contexte.

ClasseParamètresEmpreinte 4-bitTient sur
Petit1 à 1,5 B~1 GoTout portable, même sans GPU
Moyen3 à 4 B~3 GoPortable 8 Go RAM, petit GPU
Confort7 à 8 B~5 GoGPU 6 Go (juste), ou 16 Go RAM en CPU
Lourd12 à 14 B~9 GoGPU 12 Go et +
Très lourd27 à 32 B~18-20 GoStation, GPU 24 Go

Honnêteté : je ne connais pas le parc réel de tes étudiants. Avant de promettre un setup uniforme, fais lancer le diagnostic du palier 1 et relève VRAM et RAM. On calibre sur le plus petit dénominateur réel, pas supposé.

Palier 1 — Démo pédagogique tout matériel

1
Faire tourner, voir, mesurer, diagnostiquer

L'étudiant voit une IA fonctionner sur sa machine, sans réseau, et comprend ce qui se passe.

Vérifier et diagnostiquer le matériel

ollama --version
nvidia-smi        # voir le GPU NVIDIA (si pilotes installés)

Sinon : Gestionnaire des tâches > Performance > GPU (mémoire dédiée) et Mémoire (RAM).

Premier dialogue et tokens en direct

ollama run llama3.2:1b

Fais observer le défilement des tokens : le modèle ne réfléchit pas, il prédit le mot suivant, un par un. C'est le coeur de la démo.

Cas d'usage à faire tester

À copier dans l'invite >>> :

PromptCe que ça montre
Explique ce qu'est un octet à un élève de 3e.Vulgarisation : le modèle s'adapte au niveau demandé.
Combien font 17 x 23 ? Donne seulement le résultat.Piège : un petit modèle se trompe souvent en calcul. Vérifie à la calculatrice. Hallucination en direct.
Traduis en anglais technique : « régler le couple de serrage à 12 newtons-mètres ».Traduction métier, contexte industriel.

Insiste sur le calcul : un modèle de langage prédit du texte plausible, il ne calcule pas. Lien direct avec ton activité « Illusion de Vérité ».

Défi étudiantTrouve une question simple où llama3.2:1b se trompe mais où gemma3:4b répond juste. Note les deux réponses. Conclusion : la taille du modèle change la fiabilité, pas seulement la vitesse.

Mesurer (écologie et performance)

/set verbose n'est pas une commande Windows : c'est une commande interne à Ollama, à taper une fois entré dans le dialogue. Déroulé exact :

1. Lance le modèle :

ollama run gemma3:4b

2. Ollama affiche une invite >>>. 3. Tape la commande, Entrée. Confirmation :

>>> /set verbose
Set 'verbose' mode.

4. Pose une question. 5. Sous la réponse s'affiche le bloc de stats. Résultat attendu :

total duration:       4.812s
load duration:        38.7ms
prompt eval count:    18 token(s)
prompt eval duration: 412ms
prompt eval rate:     43.7 tokens/s
eval count:           96 token(s)
eval duration:        4.3s
eval rate:            22.3 tokens/s

La ligne qui compte est la dernière, eval rate : la vitesse de génération en tokens/seconde. Plus c'est haut, plus le modèle écrit vite. C'est l'indicateur à comparer entre modèles.

  • total duration : temps total de la requête.
  • load duration : chargement du modèle en mémoire (quasi nul s'il était déjà chargé).
  • prompt eval count / rate : tokens de ta question, vitesse de lecture.
  • eval count : tokens générés dans la réponse.

Démo écologie : même question sur deux modèles, on compare eval rate. Typiquement llama3.2:1b ~60-100 tokens/s (léger), gemma3:4b ~15-30 tokens/s (plus capable, plus lourd). Chiffres variables selon la machine : à mesurer en direct, pas à annoncer d'avance. Le bon réflexe n'est pas « le plus puissant » mais « le plus petit qui fait le travail ».

Pour finir la session : /set quiet coupe les stats, /bye quitte le dialogue.

Diagnostiquer les pannes

ollama list      # modèles installés
ollama ps        # modèles chargés en mémoire
ollama rm <nom>   # libérer du disque
  • Serveur muet : service Ollama non lancé, relancer l'application.
  • « model not found » : faire ollama pull <nom> d'abord.
  • Lenteur extrême : le modèle déborde de la VRAM, prendre plus petit.

Souveraineté : la preuve par la coupure réseau

Coupe le Wi-Fi, relance ollama run gemma3:4b, pose une question. Ça marche, aucune donnée ne sort. L'argument souveraineté rendu tangible.

Palier 2 — Assistant de code (Continue) matériel modeste

2
Le setup exact du prof : deux modèles dans VS Code

Un modèle léger pour l'autocomplétion, un plus costaud pour le chat. Gestion tokens/énergie, zéro clé API.

Télécharger les modèles

ollama pull qwen2.5-coder:1.5b   # autocomplétion (fill-in-the-middle)
ollama pull qwen2.5-coder:7b     # chat de code (sur ta machine)

Sur portable faible : remplace le chat 7B par qwen2.5-coder:1.5b ou gemma3:4b.

Installer puis configurer Continue

Extension « Continue » dans VS Code, puis fichier config.yaml (100 % local, aucune clé) :

name: Setup local UIMM
version: 0.0.1
schema: v1
models:
  - name: Chat Code 7B
    provider: ollama
    model: qwen2.5-coder:7b
    roles:
      - chat
      - edit
  - name: Autocomplete 1.5B
    provider: ollama
    model: qwen2.5-coder:1.5b
    roles:
      - autocomplete

Vérifie aussi que editor.inlineSuggest.enabled est à true dans VS Code.

Les cinq intentions pédagogiques, concrètement

écologieLe 1.5B tourne en continu (autocomplétion), le 7B seulement à la demande. On ne sort le calcul lourd que si nécessaire.
sécuritéAucune clé API dans cette config. Compare avec une clé cloud qui traînerait dans un fichier versionné. L'erreur classique à éviter.
disponibilitéAjoute un modèle de secours dans la liste. Si l'un ne charge pas, on bascule.
démontrerGarde ollama ps ouvert pendant que tu codes : les étudiants voient le modèle répondre en direct.
diagnostiquerPas d'autocomplétion ? Dans l'ordre : ollama list, ollama ps, inlineSuggest, nom du modèle dans config.yaml.

Palier 3 — Agent autonome (Goose) ta machine

3
Une IA qui agit sur tes fichiers

Pas seulement répondre : créer des dossiers, lire et écrire, exécuter des tâches.

Comment fonctionne un agent (la notion clé)

Un agent, c'est un LLM + des outils + une boucle. Le cycle, à dessiner au tableau :

  1. Tu donnes un objectif en langage naturel.
  2. Le LLM raisonne et choisit un outil (lire un fichier, en écrire un, lancer une commande).
  3. L'outil s'exécute et renvoie un résultat.
  4. Le LLM observe ce résultat et décide de l'étape suivante.
  5. Il recommence jusqu'à atteindre l'objectif, puis s'arrête.

La différence avec un chatbot tient en une phrase : le chatbot produit du texte, l'agent produit des actions. Le LLM décide, les outils agissent. Le MCP (Model Context Protocol) est le standard qui branche ces outils sur l'agent ; dans Goose ils s'appellent des « extensions ».

Avertissement de réalisme

Un agent fiable a besoin d'un modèle qui appelle des outils proprement. Sur petit modèle CPU, ça échoue souvent (JSON malformé). Démo sur ta machine (GPU 6 Go mini), pas sur les portables. Modèle qwen3:8b ou qwen2.5-coder:7b, objectifs simples et vérifiables. Goose est passé sous la Linux Foundation fin 2025.

Installer Goose (Windows, desktop)

  1. Page officielle des versions : github.com/block/goose/releases.
  2. Télécharge l'installeur Windows (Goose-Setup-x.x.x.exe).
  3. Installe, ouvre l'application. Prérequis : Ollama tourne (vérifie avec ollama ps).

Configurer pas à pas

  1. Paramètres (engrenage) > Provider : choisis Ollama.
  2. URL : laisse http://localhost:11434 (adresse locale, rien ne sort).
  3. Modèle : qwen3:8b (fais ollama pull qwen3:8b avant si besoin).
  4. Étanchéité : aucun provider cloud actif.
  5. Extensions : active File System, sinon l'agent ne lit ni n'écrit rien.

Sécurité : toujours un bac à sable

Un agent peut écraser ou supprimer des fichiers. Ne le lâche jamais sur tout le disque.

  • Crée 00Exp/bac_a_sable et garde tes objectifs dans ce dossier.
  • Sauvegarde ou commit Git avant une tâche qui modifie l'existant.
  • Lis ce que l'agent propose avant de valider. Il montre ses étapes, ne clique pas « oui » en aveugle.

Cinq cas d'usage progressifs

Tous dans bac_a_sable. Pour les cas 3 et 5, crée d'abord les fichiers d'exemple.

Cas 1 — Échauffement. « Crée bonjour.txt contenant la date du jour et la phrase : IA locale opérationnelle. »
Attendu : l'agent écrit le fichier. Vérifie qu'il existe et contient le texte.

Cas 2 — Générer et exécuter. « Crée fibonacci.py qui affiche la suite jusqu'à 20, puis exécute-le et montre la sortie. »
Attendu : il écrit, lance, renvoie la sortie réelle. Deux outils enchaînés : on voit la boucle.

Cas 3 — Analyse de données machine (industriel). Crée d'abord releves.csv :

machine,temperature
P1,72
P2,88
P3,65
P4,91
P5,79

« Lis releves.csv. Calcule la température moyenne. Liste les machines au-dessus de 80. Écris le tout dans rapport.md. »
Attendu : lecture, calcul, filtre, écriture. Vérifie à la main : moyenne = 79, machines > 80 = P2 et P4.

Cas 4 — Rangement de fichiers. « Range les fichiers de ce dossier dans des sous-dossiers selon leur extension : txt, py, csv. »
Attendu : il liste et déplace. Rappelle la règle du bac à sable : déplacer peut casser des chemins ailleurs.

Cas 5 — Synthèse multi-documents. Crée deux ou trois petits .md. « Lis tous les .md de ce dossier et écris une synthèse en 5 points dans synthese.md. »
Attendu : lecture multiple, agrégation, rédaction. Le cas le plus proche de ton métier de formateur.

Diagnostiquer l'agent

  • Il boucle ou rate un appel d'outil : modèle trop petit, passe à un 7B-8B.
  • Rien ne s'écrit : File System non activée, ou mauvais dossier. Précise le chemin dans le prompt.
  • Très lent : normal, chaque étape est une inférence. Réduis le modèle ou découpe la tâche.
  • Il invente au lieu de lire : exige l'outil (« lis réellement le fichier, ne devine pas »).
Défi étudiantDonne le cas 3 à l'agent. Vérifie sa moyenne à la main. S'est-il trompé ? A-t-il lu le fichier ou inventé les chiffres ? Conclusion : même un agent se contrôle. L'autonomie n'est pas la fiabilité.

Alternative honnête : le prof du post n'utilise pas Goose mais Continue. Pour un premier contact agentique, le mode agent/edit de Continue, ou Cline/Aider, suffit souvent et se débugge plus facilement. Ne complique pas si le palier 2 couvre ton besoin.

Pièges réels (pas ceux inventés par les guides)

  • Confondre runner et modèle : changer de logiciel ne règle pas un problème de mémoire. Réduis la taille du modèle.
  • Vouloir le plus gros modèle : contre-productif sur matériel modeste.
  • Recopier des noms de modèles de mémoire ou depuis un PDF : vérifie sur ollama.com/library.
  • Clé API dans un fichier versionné : en local pur tu n'en as pas. Si un jour tu ajoutes un modèle cloud de secours, mets la clé en variable d'environnement, jamais dans le dépôt.
  • Promettre un setup uniforme sans connaître le parc : mesure d'abord.

Ce que je ne sais pas, à vérifier toi-même

  • Le parc matériel réel de tes étudiants. À mesurer.
  • Les tags exacts des dernières versions Qwen3-Coder. À vérifier avant pull.
  • Le comportement de Goose sur ta machine. L'agentique locale reste plus fragile que le chat.
